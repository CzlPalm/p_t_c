# TLS Capture：README + ARCHITECTURE

## 1. 项目简介

`tls_capture` 是一个面向 Chrome / Chromium TLS 会话分析的实验性工具，目标是：

1. 在运行时捕获 TLS 密钥材料
2. 将密钥与 TCP 五元组进行关联
3. 导出可用于 Wireshark 解密的纯净 keylog 文件
4. 对比环境变量 `SSLKEYLOGFILE` 输出，验证捕获完整性

当前实现结合了三类能力：

- **Frida**：Hook Chrome `NetworkService` 内的 TLS 关键导出点
- **eBPF**：监听 `connect()` 行为，获取连接目标信息
- **用户态关联器**：把 TLS 密钥和连接五元组做 fd / 时序关联

---

## 2. 当前能力概览

### 已实现能力

- 自动检测 Chrome 版本
- 自动加载版本匹配的 Hook JSON
- 支持 `--auto` 模式自动启动 Chrome 并自动附加
- 支持 `--pid` 模式附加指定进程
- 支持 TLS 1.2 与 TLS 1.3 关键 secret 捕获
- 支持五元组注释输出
- 支持 Wireshark 无注释导出文件
- 支持环境变量 keylog 对比
- 支持退出总结统计
- 支持较稳的退出排空与 cleanup

### 当前输出类型

程序可生成两类输出：

1. **完整 keylog**
   - 包含五元组注释与密钥行
2. **Wireshark 导出 keylog**
   - 只包含标准 keylog 行，不带 `#` 注释

### 目录说明

#### `tls_capture.py`
主入口，负责：

- 参数解析
- 模式选择
- 启动 Chrome
- 启动 eBPF 子进程
- attach Frida
- 生命周期管理
- 退出清理与总结打印

#### `lib/output_writer.py`
负责输出写入：

- 完整 keylog 文件
- Wireshark 纯净导出文件

#### `lib/correlator.py`
负责五元组关联逻辑：

- 解析 `fd_tracker` 输出
- fd 精确关联
- 时序 fallback 关联
- 统计命中情况

#### `lib/version_detect.py`
负责版本与 Hook 装配：

- 检测 Chrome 版本
- 加载 `hooks/*.json`
- 将配置注入 Frida 模板
- 查找 `NetworkService` PID

#### `hooks/`
存放各 Chrome 版本对应的 Hook JSON 与 Frida 模板。

#### `ebpf/`
存放 `fd_tracker` 相关实现与编译结果。

#### `tests/`
预留测试目录。当前端到端测试主要以命令流程方式执行。

## 4. 快速开始

### 4.1 依赖准备

建议先安装 `requirements.txt` 中的依赖，并确保：

- 已安装 Frida Python 绑定
- 本机存在可用的 Chrome/Google Chrome
- 能编译并运行 `ebpf/fd_tracker`

### 4.2 编译 eBPF 组件

```bash
cd ebpf && make
```

### 4.3 推荐启动方式

为了在退出时保留 summary，推荐：

```bash
sudo $(which python3) -u tls_capture.py --auto 2>&1 | tee -i capture.log
```

如果只关心日志完整落盘，也可以：

```bash
sudo $(which python3) -u tls_capture.py --auto > capture.log 2>&1
```

### 4.4 启用环境变量对比

```bash
sudo $(which python3) -u tls_capture.py \
  --auto \
  --enable-env-keylog \
  --wireshark-export /tmp/chrome_tls_ws.log \
  2>&1 | tee -i capture.log
```

- `--no-tuple`

- `--connect-wait-timeout <SECONDS>`

- `--chrome-bin <PATH>`
  - 指定 Chrome 可执行文件
- `--user-data-dir <PATH>`
  - 指定 Chrome profile 目录
- `--verbose`
  - 输出更多调试信息

## 7. 退出总结说明

当前程序在正常 `Ctrl+C` 退出后，会输出一组总结信息，包括：

- 密钥总数
- 来源分布（`prf` / `hkdf` / `key_exp`）
- 五元组命中数
- fd 精确命中数
- 时序命中数
- connect 事件总数
- 唯一连接数
- 输出文件路径
- 环境变量对比命令

为了提高退出总结的稳定性，当前实现采用：

- 信号处理函数只设置退出标志
- 主线程统一进入 cleanup
- cleanup 前等待回调线程排空
- 关键打印使用 `flush=True`

# ARCHITECTURE

## 8. 系统总体架构

整体架构可以概括为四层：

1. **控制层**：`tls_capture.py`
2. **注入层**：Frida Hook
3. **内核观测层**：eBPF `fd_tracker`
4. **关联与输出层**：`correlator` + `output_writer`

逻辑流程如下：

```text
Chrome / NetworkService
        │
        │ (Frida Hook 捕获 TLS secret)
        ▼
   tls_capture.py
        │
        ├── _handle_key()
        │       │
        │       ├── correlator.find_connect()
        │       └── writer.write_key()
        │
        └── cleanup() / summary

fd_tracker (eBPF)
        │
        │ stderr 输出 connect 事件
        ▼
_fd_tracker_reader()
        ▼
fd_tracker_line_queue
        ▼
Correlator.parse_fd_tracker_lines()
```

---

## 9. 各模块职责

### 9.1 `tls_capture.py`

职责：

- 解析参数
- 管理全局状态
- 启动和停止 Chrome
- 启动和停止 `fd_tracker`
- 生成 Frida 脚本
- attach 到 `NetworkService`
- 接收 Frida 消息
- 汇总统计和退出清理

它是整个系统的协调中心。

### 9.2 `version_detect.py`

职责：

- 获取 Chrome 精确版本号
- 在 `hooks/` 中寻找最适合的 JSON
  - 先 exact match
  - 再 major match
- 构造最终 Hook 注入脚本
- 通过 `/proc/*/cmdline` 搜索 `NetworkService`

该模块保证“版本—Hook 配置—实际注入脚本”的闭环。

### 9.3 Frida Hook

职责：

- 在 Chrome 的 TLS 关键导出点进行 hook
- 发送：
  - `key`
  - `dbg`
  - `ready`

`key` 消息中包含：

- 密钥行内容
- 来源类型
- 进程 pid
- fd 信息（若可用）

### 9.4 `correlator.py`

职责：

- 解析 `fd_tracker` 的 connect 输出
- 以 `(pid, fd, dst_ip, dst_port)` 形式保存事件
- 提供两种匹配：
  - `fd` 精确匹配
  - 时间顺序 fallback
- 维护统计：
  - `fd_hits`
  - `time_hits`
  - `total_events`

这是五元组关联的核心。

### 9.5 `output_writer.py`

职责：

- 把五元组注释写入完整 keylog
- 把密钥行写入完整 keylog
- 同时可写入 Wireshark 导出文件
- 提供纯净导出能力

当前实现采用线程锁保证写入安全。

## 10. 关键数据流

### 10.1 密钥捕获数据流

1. Frida hook 命中 TLS 导出点
2. 发送 `key` 消息到 Python
3. `on_message()` 收到消息
4. `_handle_key()` 解析密钥内容
5. 通过 `cr_hex` 做缓存命中或实时关联
6. 输出五元组注释与密钥行

### 10.2 五元组关联数据流

1. eBPF `fd_tracker` 观测到 `connect()`
2. 输出 connect 行到 stderr
3. `_fd_tracker_reader()` 读取并送入队列
4. `Correlator.parse_fd_tracker_lines()` 解析并缓存事件
5. `_handle_key()` 调用 `find_connect()` 取最合适事件

### 10.3 退出清理数据流

1. `SIGINT` / `SIGTERM` -> `request_shutdown()`
2. `running = False`
3. 主循环退出，进入 `cleanup()`
4. 等待回调线程排空
5. detach / terminate 子对象
6. 统计并打印 summary
7. `sys.exit()`

---

## 11. 当前关联策略说明

当前五元组关联有三种来源：

1. **缓存命中**
   - 同一个 `cr_hex` 已经关联过
2. **fd 精确命中**
   - `find_connect(pid, fd)` 直接按 fd 找到 connect 事件
3. **时序命中**
   - 没有可用 fd 时，按最近未使用的 connect 事件进行 fallback

这也是 summary 中：

- `fd精确`
- `时序`
- `缓存`

三类统计的来源。

## 12. 当前实现的设计取舍

### 优点

- 结构清晰，便于调试
- 模块职责明确
- 与 Wireshark / 环境变量 keylog 兼容性好
- 支持自动化运行与汇总分析
- 退出路径已明显稳于早期版本

### 当前局限

- 仍主要面向 Chrome `NetworkService`
- 五元组关联在 `fd=-1` 场景下会更多依赖时序 fallback
- `tests/` 目录还未形成完整自动化测试代码
- 生命周期控制仍主要集中在单文件主程序中
