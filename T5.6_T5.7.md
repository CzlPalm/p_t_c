# T5.6 进程生命周期管理分析 与 T5.7 端到端测试

## 一、T5.6：进程生命周期管理

本文基于当前代码状态分析 `tls_capture.py` 的进程生命周期管理实现，说明该任务已经完成到什么程度、由哪些模块协同实现、以及当前设计的关键点。

---

## 1. 生命周期涉及的核心对象

当前主程序维护了以下全局状态：

- `sessions`
  - 保存 Frida attach 后的 session，退出时统一 `detach`
- `fd_tracker_proc`
  - eBPF `fd_tracker` 子进程句柄
- `fd_tracker_reader_thread`
  - 读取 `fd_tracker` stderr 的线程
- `fd_tracker_line_queue`
  - `fd_tracker` 输出队列，供关联器消费
- `spawned_proc`
  - `--auto` 模式下自动拉起的 Chrome 进程句柄
- `running`
  - 主循环退出标志
- `cleanup_started`
  - 防止重复清理
- `inflight_callbacks`
  - 统计正在执行的 Frida 消息回调数，退出前用于排空

这些对象共同组成了当前的生命周期控制面。

---

## 2. 生命周期阶段划分

### 2.1 启动前：参数解析与运行模式选择

程序入口在 `main()`：

- 调用 `parse_args()` 解析参数
- 支持两种 attach 方式：
  - `--auto`：自动启动 Chrome 并自动附加
  - `--pid`：附加指定 PID
- 支持：
  - 自定义输出文件
  - Wireshark 导出
  - 是否启用五元组关联
  - 自定义 `user-data-dir`
  - 自定义 Chrome 路径
  - 是否同时开启环境变量 `SSLKEYLOGFILE`

这部分完成了生命周期的“配置阶段”。

### 2.2 版本检测与 Hook 配置装配

`resolve_hook_script()` 负责：

1. 调用 `detect_chrome_version()` 检测 Chrome 版本
2. 调用 `load_config()` 从 `hooks/` 中加载对应 JSON
3. 调用 `build_hook_script()` 生成最终 Frida hook 脚本

这部分确保 attach 前，Hook 配置与浏览器版本匹配。

### 2.3 eBPF 子进程启动

`start_fd_tracker()` 负责 `fd_tracker` 生命周期的启动部分：

1. 检查 `ebpf/fd_tracker` 是否存在
2. `subprocess.Popen()` 启动子进程
3. 建立 `fd_tracker_line_queue`
4. 启动 `_fd_tracker_reader()` 线程读取输出
5. 通过 `_wait_fd_tracker_ready()` 等待 ready 标志
6. 启动关联器线程 `correlator.parse_fd_tracker_lines()`

这意味着当前实现已经完成：

- eBPF 子进程创建
- 子进程 ready 检测
- 子进程输出转队列
- 关联引擎后台消费

### 2.4 Chrome 进程管理

#### 手动模式

`attach_existing()`：

- 在指定 `user-data-dir` 下查找 `NetworkService`
- 成功后 attach 到 NetworkService

#### 自动模式

`auto_mode()` -> `spawn_chrome_as_user()` + `attach_existing()`：

- 在 root 启动脚本的情况下，利用 `SUDO_UID/SUDO_GID` 回落到原始用户身份
- 补齐 GUI 相关环境变量：
  - `DISPLAY`
  - `XAUTHORITY`
  - `XDG_RUNTIME_DIR`
  - `WAYLAND_DISPLAY`
  - `HOME`
  - `USER`
  - `LOGNAME`
- 可选注入 `SSLKEYLOGFILE`
- 通过 `preexec_fn` 降权，以原始用户身份启动 Chrome
- 后续等待 `NetworkService` 出现并自动 attach

这部分说明 T5.6 中“自动拉起 + 自动附加 + 退出统一回收”的核心链路已经建立。

### 2.5 Frida 回调运行期管理

`on_message()` 为每个 attach 的 PID 创建消息处理器。

关键点：

- 所有消息回调开始时 `inflight_callbacks += 1`
- 处理结束后在 `finally` 中 `inflight_callbacks -= 1`
- `key` 消息进入 `_handle_key()`
- `dbg` / `ready` 直接打印状态

`_handle_key()` 完成：

1. 统计密钥数量与来源
2. 提取 `CLIENT_RANDOM` / TLS 1.3 secret 对应的 `cr_hex`
3. 优先从 `cr_to_tuple` 命中缓存
4. 未命中则通过 `correlator.find_connect()` 做关联
5. 关联成功时写入五元组注释
6. 写入完整 keylog
7. 输出彩色控制台日志

### 2.6 退出路径与清理阶段

当前退出流程已经是“主线程统一清理”模型：

#### 信号阶段

`request_shutdown()`：

- 收到 `SIGINT` / `SIGTERM` 后
- 只设置 `running = False`
- 打印退出提示
- 不在信号处理函数中做重型清理

#### 主循环阶段

主线程运行：

- `while running: time.sleep(0.1)`
- 当 `running` 变为 `False` 时跳出
- 再进入 `cleanup()`

#### cleanup 阶段

`cleanup()` 当前实现包含：

1. 幂等保护：`cleanup_started`
2. 停止主循环：`running = False`
3. 等待回调线程排空：最多 `SHUTDOWN_DRAIN_TIMEOUT = 0.5s`
4. `detach` 所有 Frida session
5. 向 `fd_tracker_line_queue` 投递 `None` 结束关联线程
6. `terminate/kill` `fd_tracker_proc`
7. `terminate/kill` `spawned_proc`
8. 汇总统计：
   - 密钥总数
   - 来源分布
   - 五元组命中数
   - fd 精确命中数
   - 时序命中数
   - connect 总事件数
   - 唯一连接数
9. 导出 Wireshark 纯净文件（如启用）
10. 打印对比命令并退出

# 二、T5.7：端到端测试

本文给出当前版本的端到端测试方法，覆盖以下目标：

1. 捕获结果与环境变量 `SSLKEYLOGFILE` 进行 `diff` 比对
2. 统计五元组命中率
3. 检查 Wireshark 导出文件不含注释行
4. 验证退出时 summary 正常输出

---

## 1. 测试准备

### 1.1 编译 eBPF 组件

在项目根目录执行：

```bash
cd ebpf && make
```

### 1.2 确认 Hook 配置存在

检查 `hooks/` 下是否有与 Chrome 版本匹配的 JSON。

### 1.3 清理旧文件

```bash
rm -f capture.log /tmp/chrome_tls_capture.log /tmp/chrome_tls_ws.log /tmp/chrome_sslkeys_env.log
```

---

## 2. 测试一：自动启动 + 环境变量对比

### 2.1 启动命令

建议使用 `tee -i`，避免 `Ctrl+C` 导致日志记录端提前退出：

```bash
sudo $(which python3) -u tls_capture.py \
  --auto \
  --enable-env-keylog \
  --wireshark-export /tmp/chrome_tls_ws.log \
  2>&1 | tee -i capture.log
```

### 2.2 访问测试站点

启动完成后，在 Chrome 中访问多个 HTTPS 站点，例如：

- `https://www.google.com`
- `https://www.baidu.com`
- `https://www.bing.com`
- 任意一个会触发多域名请求的站点

建议至少打开：

- 1 个简单站点
- 1 个重资源站点
- 1 个存在重定向或多 CDN 请求的站点

### 2.3 正常退出

在脚本所在终端按 `Ctrl+C`。

### 2.4 检查退出总结

在 `capture.log` 结尾确认存在：

- `[*] 收到退出信号，准备清理并输出总结...`
- `[*] 等待回调线程排空...`
- `[*] 开始执行清理...`
- `[*] 密钥: ... 条`
- `五元组命中: ...`
- `connect 事件: ...`
- `[*] 输出文件: ...`

如果这些行都出现，说明端到端生命周期测试通过。

---

## 3. 测试二：与环境变量 keylog 做 diff 比对

### 3.1 测试目标

验证 Frida 捕获的纯密钥集合与 `SSLKEYLOGFILE` 输出一致或基本一致。

### 3.2 对比命令

程序退出时已经打印了参考命令，也可以手动执行：

```bash
diff <(grep -v "^#" /tmp/chrome_tls_capture.log | sort) \
     <(grep -E "^(CLIENT|SERVER|EXPORTER)" /tmp/chrome_sslkeys_env.log | sort)
```

### 3.3 判定标准

- **理想结果**：`diff` 无输出
- **允许现象**：少量差异需要结合访问时机分析
- **失败判定**：大量缺失，说明 Hook 链路、版本配置或退出排空可能有问题

### 3.4 记录建议

记录以下信息：

- Chrome 版本
- Hook JSON 名称
- 访问站点列表
- Frida key 总数
- 环境变量 key 总数
- `diff` 是否为空

---

## 4. 测试三：五元组命中率检测

### 4.1 测试目标

验证 TLS 密钥是否能成功关联到五元组。

### 4.2 观察位置

程序退出 summary 中有：

- `五元组命中: X/Y`
- `关联方式: fd精确=A  时序=B  缓存=C`
- `connect 事件: N  唯一连接: M`

### 4.3 计算方式

命中率计算为：

\[
命中率 = X / Y
\]

其中：

- `X = tuple_hits`
- `Y = keycount`

### 4.4 判定建议

- **高命中率**：大部分 key 都带 `->dst_ip:dst_port`
- **中等命中率**：说明 fd 精确命中有限，但时序 fallback 仍工作正常
- **低命中率**：优先排查
  - `fd_tracker` 是否正常启动
  - `connect` 事件是否足够
  - `--no-tuple` 是否误开启
  - `connect_wait_timeout` 是否过短

### 4.5 建议记录

记录：

- `五元组命中: X/Y`
- `fd精确` 命中数
- `时序` 命中数
- `缓存` 命中数
- `connect 事件` 总数

## 5. 测试四：Wireshark 导出文件无注释行

### 5.1 测试目标

验证 `--wireshark-export` 生成的文件不包含 `# five_tuple=` 注释行，只包含标准 keylog 内容。

### 5.2 运行方式

启动时加上：

```bash
--wireshark-export /tmp/chrome_tls_ws.log
```

### 5.3 检查命令

检查是否含注释：

```bash
rg '^#' /tmp/chrome_tls_ws.log
```

### 5.4 判定标准

- **正确**：无输出
- **错误**：只要出现以 `#` 开头的行，就说明导出逻辑有问题

### 5.5 进一步检查

确认至少有 key 行：

```bash
rg '^(CLIENT|SERVER|EXPORTER)' /tmp/chrome_tls_ws.log
```

若该命令能匹配到多条记录，说明导出文件内容有效。

## 6. 测试五：完整文件结果检查

### 6.1 完整 keylog 文件应包含两类行

`/tmp/chrome_tls_capture.log` 应包含：

1. 注释行：

```text
# five_tuple=tcp:src_ip:src_port->dst_ip:dst_port pid=... fd=...
```

2. 密钥行：

```text
CLIENT_RANDOM ...
CLIENT_HANDSHAKE_TRAFFIC_SECRET ...
SERVER_HANDSHAKE_TRAFFIC_SECRET ...
CLIENT_TRAFFIC_SECRET_0 ...
SERVER_TRAFFIC_SECRET_0 ...
EXPORTER_SECRET ...
```

### 6.2 抽样检查

可通过如下命令快速查看：

```bash
rg '^(# five_tuple|CLIENT|SERVER|EXPORTER)' /tmp/chrome_tls_capture.log
```
