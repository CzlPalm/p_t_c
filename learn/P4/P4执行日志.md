### 3、**任务 T4.0.3**：在 Ghidra 中定位 ssl_st → rbio → fd 路径（1–2d）：
代码保存为watchdog_attach_v7.py，
**JS 层（三层扫描）：** 去掉了 `_fdScanned` 限制，每次 HKDF 命中都执行 `scanFdLayers()`，同时扫描 `ssl` / `s3` / `sub` 三层，每层读 `0x300` 字节，以 `Int32Array` 形式传回 Python。
**Python 层（动态匹配 + 统计确认）：**
- `get_known_fds(pid)` 每次从 `/proc/{pid}/fd` 实时读取当前 socket fd 集合，不依赖手动 lsof
- `record_fd_candidates()` 跨多次 HKDF 调用累计命中次数，某偏移在某层命中 ≥3 次即自动确认
- `fd_to_tuple()` 通过 socket inode → `/proc/net/tcp` 反查五元组，无需任何手动操作
- 退出时将发现结果写入 `/tmp/fd_offset_discovery.json`，格式为 `{"layer", "offset", "confirmed"}`：
**实验逻辑链**：工具链准备 → uprobe 挂载验证 → Frida HKDF/PRF hook 建立 → fd 动态探测（v6 系列三层扫描+双重稳定性验证）→ BIO 指针链完整探测→ wbio 交叉验证确认结果。

**方法选择的核心理由**：Chrome stripped 二进制使 Ghidra 静态分析路径成本极高，而 HKDF hook 点天然持有 `ssl_ptr`，结合 `/proc/{pid}/fd` 的已知 fd 集合，可以用"已知值反查偏移"的动态方式绕过符号缺失问题。

**最终结论**：`ssl_st.rbio = ssl_ptr + 0x240`，`bio_st.num = BIO* + 0x03c`，wbio 交叉验证通过，结果可信，在同版本 Linux x86-64 Chrome 上完全可复用。

**结果：
**`ssl+0x240` 在所有 HKDF 调用中都是 NULL。** T4.0.3 测出的偏移 0x240 是对 `ssl_st` 结构体有效的，但 **HKDF 函数的 `args[0]` 不是 `ssl_st*`**——它是 HKDF 的内部上下文对象，结构体布局完全不同。T4.0.3 的探测是在 HKDF 触发后对 ssl_ptr 做扫描，但那个 ssl_ptr 是从调用栈上层传下来的，不等于 HKDF 函数自身的第一个参数。

这个问题在当前架构下无法简单修复——需要在 HKDF 的调用者（如 `FUN_049836b0`）中 hook 拿到真正的 `ssl_st*`，再传递给 HKDF hook，复杂度高且版本脆弱。
