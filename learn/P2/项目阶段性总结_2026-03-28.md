# TLS 密钥捕获与五元组关联系统 — 项目阶段性总结

**更新日期**: 2026-03-28  
**当前阶段**: P3 已完成验收，准备进入 P4  
**目标平台**: Linux x86-64, Chrome (BoringSSL)

---

## 一、项目目标

基于 TLSKeyHunter 论文（Baier & Lambertz, DFRWS APAC 2025）的核心技术，构建一套多浏览器 TLS 密钥自动捕获与网络五元组关联系统。核心创新：预先离线分析各版本浏览器二进制，提取密钥派生函数（PRF/HKDF）的字节指纹和偏移量，构建持久化指纹数据库。运行时通过精确版本号命中数据库（零延迟），或回退至内存指纹扫描动态定位。

最终交付物：输入浏览器 PID → 输出带五元组注释的 NSS Key Log 格式密钥文件，可直接导入 Wireshark 解密。

---

## 二、技术基础

**论文**: "All your TLS keys are belong to Us" (FSI:DI, 2025)  
**GitHub**: https://github.com/monkeywave/TLSKeyHunter  
**核心思路**: TLS 规范强制要求固定标签字符串（"master secret"/"c hs traffic" 等），通过静态分析定位这些标签在二进制中的引用，前向数据流追踪找到密钥派生函数，提取字节指纹用于运行时 Hook。

**当前分析目标**: Chrome (Linux x86-64)，TLS 库为 BoringSSL  
**Ghidra imageBase**: `0x00100000`

#### TLS 1.3 密钥内存偏移量表

| label | NSS 标签 | 输出偏移 | 长度字节偏移 |
|-------|----------|---------|-------------|
| c e traffic | CLIENT_EARLY_TRAFFIC_SECRET | ssl_ptr + 0x51 | *(byte*)(ssl_ptr + 0x81) |
| c hs traffic | CLIENT_HANDSHAKE_TRAFFIC_SECRET | ssl_ptr + 0x82 | *(byte*)(ssl_ptr + 0xb2) |
| s hs traffic | SERVER_HANDSHAKE_TRAFFIC_SECRET | ssl_ptr + 0xb3 | *(byte*)(ssl_ptr + 0xe3) |
| c ap traffic | CLIENT_TRAFFIC_SECRET_0 | ssl_ptr + 0xe4 | *(byte*)(ssl_ptr + 0x114) |
| s ap traffic | SERVER_TRAFFIC_SECRET_0 | ssl_ptr + 0x115 | *(byte*)(ssl_ptr + 0x145) |
| exp master | EXPORTER_SECRET | *(ssl_ptr[0]+0x30)+0x182 | *(byte*)(*(ssl_ptr[0]+0x30)+0x1b2) |
| res master | (不写 keylog) | ssl_ptr[0xfa]+10 | — |

#### 结构体偏移量

```
ssl_st:
  [0x00] → SSL3_STATE* s3 (第一个字段)

SSL3_STATE:
  client_random: *(*(ssl_ptr) + 0x30) + 0x30 → 32字节  ★ 注意是 +0x30 不是 +0x10
  server_random: *(*(ssl_ptr) + 0x30) + 0x10 → 32字节

TLS 1.2 Extended Master Secret 标志:
  *(byte*)(ssl_ptr + 0x812) & 0x02
```

**⚠️ 关键教训**: 初始分析时 client_random 偏移写成了 `+0x10`（错误），经过三个连接的 RAW 内存 dump + SSLKEYLOGFILE 逐字节交叉比对，最终确认正确路径是 `+0x30`。这个错误导致了 P3 初期所有密钥无法解密。

### P2：运行 TLSKeyHunter，提取密钥指纹并交叉验证 ✅ 已完成

**HKDF 对比结果**: 手工提取与 TLSKeyHunter 自动提取字节级完全一致。手工版多了尾部 8 字节（`84 C0 0F 84 C9 00 00 00`，即 `TEST AL,AL; JZ` 条件跳转），不影响匹配。RVA `0x048837E0` 完全一致。

**PRF 对比结果**: TLSKeyHunter 自动识别**失败**。原因有三:
1. "master secret" 是 "extended master secret" 的子串，XREF 追踪断链
2. HKDF 有 .rodata 字节模式回退路径，PRF 没有
3. 新版 BoringSSL 的 14 参数 PRF 函数超出验证启发式范围

**结论**: PRF 指纹需要手工提取或修改 TLSKeyHunter 源码增加 .rodata 回退，这是 P8 自动化阶段需要解决的问题。

## 五、P4 阶段计划（五元组关联）

**推荐架构**: Frida（密钥提取）+ eBPF（五元组关联）混合方案

**原因**: eBPF uprobe 可以挂载到 RVA 读取寄存器，但 client_random 的三级指针解引用 + label 字符串比较在 eBPF 512 字节栈限制下实现复杂。五元组关联是 eBPF 的天然优势（内核态 `inet_sock_set_state` tracepoint + BPF Map）。

**P4 任务拆解**:
1. 在 Ghidra 中定位 SSL_set_fd 或从 ssl_st 直接读取 BIO→fd 偏移量
2. 安装 BCC/libbpf，编写 eBPF 程序监控 Chrome 的 TCP 连接事件
3. Frida 捕获密钥时同时提取 fd（从 ssl_st 结构体），通过 BPF Map 与五元组关联
4. 输出格式：`# five_tuple=tcp:192.168.1.5:54321→142.250.80.46:443\nCLIENT_HANDSHAKE_TRAFFIC_SECRET ...`

**SSL_set_fd 定位状态**: 在 Ghidra 中搜索 `no BIO` / `BIO_new_socket` 均未找到（Chrome stripped binary）。替代路径:
- 搜索 `BIO_NOT_SET`（地址 `025ca0f5`）的 XREF，从 `ssl->rbio == nullptr` 判断推断 rbio 偏移
- 或在 Frida HKDF onEnter 中用内存扫描探针找 fd
- 或先用时序 fallback（eBPF tcp_connect 事件时间戳与 HKDF 触发时间匹配）

## 六、原计划调整建议

| 原计划 | 调整 | 原因 |
|--------|------|------|
| P4: eBPF 完全取代 Frida | **改为混合架构** | eBPF 用户态多级指针解引用过于复杂 |
| P3→P4 直接衔接 | **P3.5: 补充 ssl_log_secret Hook** | 解决 4% Session Ticket 漏捕 |
| P8: TLSKeyHunter 批量自动化 | **需先修改 TLSKeyHunter 源码** | PRF 自动识别对 Chrome 失败，需增加 .rodata 回退路径 |
| 数据库 Schema | **新增 evp_md_size_offset 字段** | SHA-384 动态密钥长度需要此信息 |
| 数据库 Schema | **新增 client_random_path 字段** | 不同版本偏移可能变化，需存储完整路径 |
