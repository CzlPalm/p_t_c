## 分析：TLSKeyHunter PRF 识别失败的根本原因

### 一、先确认 HKDF 交叉验证结果

HKDF 部分**前大部分完全一致**，逐字节比对：

```
手工提取: 55 48 89 E5 41 57 41 56 41 55 41 54 53 48 81 EC 98 00 00 00 ...
TLSKeyHunter: 55 48 89 E5 41 57 41 56 41 55 41 54 53 48 81 EC 98 00 00 00 ...

结果: ✅ 字节级大部分完全一致，RVA 0x048837E0 也完全一致
```
部分解释
```
4C 89 45 88 → MOV [RBP-0x78], R8 ← param_13(label_len) 保存自 R8 
49 89 CC → MOV R12, RCX ← R12 = label 指针 (param_12) 自 RCX 
49 89 D7 → MOV R15, RDX ← R15 = HKDF ctx (param_11) 自 RDX 
48 89 F3 → MOV RBX, RSI ← RBX = ★ output buf (param_10) 自 RSI 
49 89 FE → MOV R14, RDI ← R14 = ssl_ptr (param_9) 自 RDI 
``` 所以 Frida args[] 对应关系（Frida 只追踪整数寄存器，按 RDI/RSI/RDX/RCX/R8/R9 顺序）： ``` 
args[0] = RDI = ssl_ptr 
args[1] = RSI = output_buf ← 密钥写入地址 
args[2] = RDX = HKDF ctx 
args[3] = RCX = label 字符串指针 
args[4] = R8 = label 长度 
args[5] = R9 = hash 算法
```

---

### 二、PRF 失败的根本原因：三层叠加问题

**日志中的失败链**：

```
尝试1: 直接搜索 "master secret"        → 未找到
尝试2: 搜索 "extended master secret"   → Error（不是未找到，是报错）
尝试3: Stack String Search             → 未找到
结论: PRF 识别失败
```

**原因一：搜索策略差异（最核心）**

HKDF 识别有 `.rodata` 字节模式回退路径，PRF 识别没有：

```
HKDF 识别流程：
  直接字符串搜索失败
       ↓
  ★ 启动 .rodata 十六进制模式扫描  ← PRF 没有这个回退！
       ↓
  成功定位 025ccec0 → 找到函数

PRF 识别流程：
  直接字符串搜索失败
       ↓
  尝试 "extended master secret" → Error（代码异常，不是正常失败）
       ↓
  Stack String Search → 找不到（搜索的是运行时栈构造的字符串）
```

**原因二：`"master secret"` 是 `"extended master secret"` 的子串**

从 Ghidra 分析结果可以看到两者在内存中相邻：

```
0x019091a5  "extended master secret"  (22字节)
0x019091ae  "master secret"           (13字节)
             ↑
             这个地址正好是 "extended master secret" 字符串内部的偏移 +9 处
```

TLSKeyHunter 在搜索 "master secret" 时命中了子串，**XREF 追踪失败**——因为代码实际引用的是 `0x019091a5`（完整字符串），而不是 `0x019091ae`（子串偏移），导致数据流分析断链。

**原因三：新版 BoringSSL 的 PRF 结构超出 TLSKeyHunter 的启发式范围**

从我们的反编译可以看到 `FUN_0a32d4b0` 有 14 个参数（含 XMM 寄存器传参），这种调用约定在 TLSKeyHunter 的候选函数验证逻辑中可能被误判为 wrapper 而被过滤掉。

---

### 三、这是 TLSKeyHunter 源码的已知局限

对照论文 Table 1 的结论——BoringSSL 是 **Unified PRF**，但 TLSKeyHunter 对 BoringSSL PRF 的识别在新版本 Chrome 上实际失效。这属于论文 Section 8.5 明确指出的局限：

> _"自动识别 PRF 参数仍需人工介入"_
