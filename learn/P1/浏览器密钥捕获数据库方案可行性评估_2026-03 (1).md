 # TLSKeyHunter 论文深度分析与浏览器密钥数据库方案评估

## 一、论文核心技术分析

### 1.1 整体架构：两阶段工作流

论文提出的方案分为两个互补阶段：

**阶段一：静态分析识别阶段（离线）**
- 输入：TLS 库的二进制文件
- 工具：Ghidra（v11.1.2）作为反汇编器
- 输出：Key-Derivation Fingerprint（密钥派生指纹）

**阶段二：动态提取阶段（在线/实时）**
- 输入：上阶段产出的指纹
- 工具：Frida（v16.5.2）动态插桩
- 输出：TLS 会话密钥（NSS keylog 格式，可直接导入 Wireshark）

---

### 1.2 关键技术：TLS 标签驱动的函数定位行 TLSKeyHunter，提取密钥指纹并交叉验证

这是整篇论文最核心的洞察，值得深入理解：

**TLS 协议强制要求固定标签字符串**

TLS 规范（RFC 5246 / RFC 8446）规定了密钥派生必须使用的固定字符串标签：

| 版本 | 函数 | 标签示例 |
|------|------|----------|
| TLS 1.2 | PRF | `"master secret"`, `"key expansion"` |
| TLS 1.3 | HKDF-Expand-Label | `"c hs traffic"`, `"s hs traffic"`, `"c ap traffic"`, `"s ap traffic"` |

这些标签**不可缺失**——缺少它们会导致与其他实现不兼容。因此任何合规的 TLS 实现，无论如何编译优化，都必须在二进制中保留这些字节序列，这就给了静态分析一个可靠的"锚点"。

**前向数据流分析（Forward Data Flow Analysis）**

找到标签字符串后，论文用 Ghidra 的 P-Code 中间表示进行前向数据流跟踪：

```
标签字符串地址  →  寄存器/栈传递  →  函数调用参数  →  目标函数（KDF）
```

P-Code 是架构无关的中间表示，这使得分析可以跨 x86/ARM/ARM64 等架构统一进行。

**包装函数（Wrapper）识别**

真正的 KDF 不会直接操作标签字符串（不会调用 `strcpy`/`strlen`），且返回值通常是成功/失败标志而非构造对象。论文通过这两个启发式规则区分 wrapper 和真正的 KDF。

---

### 1.3 Key-Derivation Fingerprint 生成

找到 KDF 后，提取其函数序言（prologue）的字节序列作为指纹：

```
起始位置：函数入口点
结束条件：遇到第一个非调用类分支指令（跳转/条件跳转）
最小长度：若 < 32 字节，则继续到下一个分支
```

这种方式参考了 IDA Pro 的 FLIRT 技术，16–32 字节通常足以唯一标识一个函数。最终输出：
- **字节序列**（如 `FF 43 02 D1 FD 7B 05 A9 F8 5F 06 ...`）
- **在目标库中的偏移量（RVA）**

---

### 1.4 Frida Hook 实现密钥提取

**TLS 1.2 场景**

Hook PRF 函数，拦截：
- `master_secret`（48 字节固定长度）
- `client_random`（PRF 的显式输入参数，易于提取）

**TLS 1.3 场景**

Hook HKDF-Expand-Label，分多次拦截：
- `client_handshake_traffic_secret`
- `server_handshake_traffic_secret`
- `client_application_traffic_secret_N`
- `server_application_traffic_secret_N`

注意：TLS 1.3 的 HKDF 中 `client_random` **不是**密钥派生的输入，部分库（BoringSSL、GnuTLS 等）会通过 session 结构体传递它，而 Botan/LibreSSL/Mbed TLS/Schannel 等不传，需要从 ClientHello 报文中单独提取后进行时序关联。

**密钥输出格式（NSS Key Log Format）**

```
CLIENT_RANDOM <client_random_hex> <master_secret_hex>  # TLS 1.2
CLIENT_HANDSHAKE_TRAFFIC_SECRET <cr_hex> <secret_hex>  # TLS 1.3
```

**步骤三：大版本 Fingerprint 内存扫描 ⚠️ 可行但有工程难点**

这步是整个方案中技术风险最高的，需要重点分析：

在 254MB 的 Chrome 进程空间中搜索 32 字节序列是完全可行的，但要处理几个细节。首先是搜索范围问题，不应全内存扫描，只需扫描 Chrome 主模块的 `.text` 段，可以通过 `/proc/PID/maps`（Linux）或 `VirtualQuery`（Windows）定位。其次是误匹配问题，32 字节指纹在 Chrome 里随机碰撞概率极低，但若真的有多个匹配，可以用第二个 HKDF 指纹交叉验证。

```python
def scan_for_fingerprint(pid, fingerprint_bytes):
    # 1. 定位 chrome.dll/.so 在内存中的基地址
    base_addr = get_module_base(pid, "chrome")
    # 2. 读取 .text section
    text_section = read_memory_region(pid, base_addr, text_size)
    # 3. Boyer-Moore 搜索，254MB 搜索通常 < 50ms
    offset = text_section.find(fingerprint_bytes)
    if offset != -1:
        return base_addr + offset  # 动态 RVA
```

### 3.3 主要工程挑战与解决思路

**挑战一：Chrome 更新频率高**

Chrome 大约每 4-6 周发布一个稳定版本，但其实每个大版本还有多个小版本更新（patch）。好消息是：小版本更新（如 `124.0.6367.118 → 124.0.6367.119`）通常只修安全漏洞，**不会触及 BoringSSL 的 KDF 函数**，因此同一大版本的 PRF/HKDF 指纹大概率相同。

解决方案是建立自动化流水线：

```
GitHub Actions / 自建 CI
├── 监控 Chrome Releases RSS
├── 触发: 新版本出现
│   ├── 下载 chrome.dll (Windows) / libchrome.so (Linux)
│   ├── 运行 TLSKeyHunter（Ghidra headless 模式）
│   ├── 提取 PRF + HKDF 指纹
│   └── 写入数据库，打 tag
└── 推送更新
```

Ghidra 支持 headless 模式（`analyzeHeadless`），可以完全自动化。论文数据显示 Chrome 124 的 TLSKeyHunter 分析耗时约 111 秒（主要是 Ghidra 预处理的 73144 秒），但这是离线预处理，完全可以接受。

**挑战二：多平台支持**

Chrome 在 Windows/Linux/macOS/Android 上都有发布。论文已经验证了 ARM/ARM64/x86/x86-64 均可工作，但每个平台需要单独的二进制和指纹。对应 BoringSSL 在不同平台的编译产物：

| 平台 | 目标文件 | 架构 |
|------|----------|------|
| Windows x64 | `chrome.dll` | x86-64 |
| Linux x64 | `libchrome.so` | x86-64 |
| Android ARM64 | `libchrome.so` | ARM64 |
| macOS | `Google Chrome Framework` | x86-64 / ARM64 |

**挑战三：Frida Hook 需要管理员权限**

这是运行时的根本限制，不是数据库方案本身的问题，但需要在工具设计时说明。在 Linux 上通常需要 `ptrace` 权限，Windows 上需要 SeDebugPrivilege。

### 3.4 BoringSSL（Chrome 的 TLS 库）特有分析

Chrome 使用的是谷歌维护的 BoringSSL 分支，而非标准 OpenSSL。从论文结果看，BoringSSL 的表现是所有测试库中最好的之一：PRF 和 HKDF 的 secret 提取以及 client_random 提取全部成功（Table 2 全是 ✓）。

# 多浏览器支持、五元组关联与 eBPF Hook 深度评估

## 一、多浏览器可行性分析**Chromium 系浏览器**（Chrome、Edge、Brave、Opera、Vivaldi）全部使用 BoringSSL，这意味着同一 Chromium 版本号下，所有这些浏览器的 BoringSSL 指纹**完全相同**。Edge 的 Chromium 版本与 Chrome 同步发布，数据库可以直接共享，仅需在入口处区分读取版本号的路径（Edge 的 `Last Version` 文件路径不同）。Brave 等第三方 Chromium 分支则需要单独构建，因为他们的发布节奏与上游有偏差。

**Firefox** 使用 Mozilla 维护的 NSS 库。论文 Table 1 和 Table 2 明确显示 NSS 的 PRF 和 HKDF 均提取成功（包括 client_random），且 NSS 是 Non-Unified PRF（有两套实现），TLSKeyHunter 会检测到多个交叉引用并分别提取。Firefox 需要独立的指纹数据库，但技术难度与 Chrome 相当。

**Safari** 是最大的障碍。论文直接将 CoreTLS/SecureTransport 标记为 N/A 并排除（已弃用）。现代 Safari 在 macOS 上使用私有 TLS 栈，加上苹果 SIP（System Integrity Protection）会阻止向 Safari 进程注入任何动态插桩工具。这条路在不越狱或禁用 SIP 的前提下基本不可行。

## 二、五元组关联：需要新增哪些函数 Hook

这是三个问题中技术难度最高的部分，需要在论文原有框架之外做实质性扩展。

### 2.1 问题的本质

论文目前提取的信息是：`client_random → 密钥材料`。而你需要的是：`(srcIP, srcPort, dstIP, dstPort, proto) ↔ client_random ↔ 密钥材料`。

中间的桥梁是**在 TLS 握手发生时，TLS Session 对象与 Socket 文件描述符（fd）的绑定关系**。### 2.2 需要新增的具体函数及技术路径

**方案 A（推荐）：Hook `SSL_set_fd` 建立映射表**

BoringSSL 和 NSS 都暴露了 `SSL_set_fd(SSL* ssl, int fd)` 这个函数，它在 TLS 会话与 socket 绑定时被调用。Hook 这个函数可以在 TLS 握手发生之前，建立 `{SSL 对象地址} → fd` 的映射。在已有的 PRF/HKDF Hook 触发时，通过 SSL 对象指针查表，再用 fd 查询 `/proc/net/tcp` 或调用 `getpeername(fd)` 即可还原五元组。

这个函数有导出符号，不需要像 KDF 一样做指纹识别，直接按名称 Hook 即可，工程量很小。

**方案 B（更深，更稳）：读 SSL 结构体内的 fd 字段**

在 PRF/HKDF 被 Hook 时，函数的某个参数（通常是第一个参数）是 SSL 上下文指针。BoringSSL 的 `ssl_st` 结构体内部有 `rbio`/`wbio` 字段，BIO 对象里存着 fd。需要对特定版本逆向确认偏移量，但这个偏移在小版本更新中通常稳定。好处是完全不依赖 `SSL_set_fd` 是否被调用，适合更复杂的场景（如 QUIC over UDP）。

**需要新增到指纹数据库的内容**：

```json
{
  "124.0.6367.118": {
    "prf_rva": "...",
    "hkdf_rva": "...",
    "ssl_set_fd_rva": "...",       // 新增，通常有符号直接找
    "ssl_st_fd_offset": 0x58,     // 新增，ssl_st 内 fd 字段偏移
    "ssl_st_client_random_offset": 0x120  // 新增，如需从结构体直接读
  }
}
```

前两个 RVA（prf/hkdf）用 TLSKeyHunter 自动提取，后两个偏移量目前需要半自动处理——`SSL_set_fd` 有符号所以直接获得，`ssl_st` 内的 fd 偏移可以通过对 BoringSSL 源码编译带调试符号版本后自动提取，不需要人工逆向每个版本。

## 三、eBPF 替代 Frida/ptrace 的可行性与优势

### 3.1 架构对比### 3.2 eBPF uprobe 的核心技术路径

论文在相关工作章节提到了 Valadon (2022) 的 eBPF 方案，但指出其"依赖调试符号或导出函数"。你的数据库方案恰好解决了这个限制——有了预计算的 RVA，uprobe 可以精确挂载到任意偏移，不再依赖符号。

**挂载 uprobe 的方式**：


**关键限制**：eBPF 程序存在栈大小上限（512 字节）和指令数上限（旧内核 4096 条，5.2+ 后 100 万条），读取复杂嵌套结构体需要使用 `bpf_probe_read_user` 链式调用，比 Frida 的 JS 脚本繁琐。

### 3.3 五元组 + eBPF 的天然优势

eBPF 在五元组关联上有 Frida 无法比拟的优势——可以同时 Hook 用户态函数和内核态网络事件，用同一套 BPF Map 关联它们：

```c
// uprobe 1：SSL_set_fd → 建立 {ssl_ptr → fd} 映射
SEC("uprobe/.../chrome:SSL_SET_FD_RVA")
int hook_ssl_set_fd(struct pt_regs *ctx) {
    u64 ssl_ptr = PT_REGS_PARM1(ctx);
    int fd = (int)PT_REGS_PARM2(ctx);
    bpf_map_update_elem(&ssl_to_fd, &ssl_ptr, &fd, BPF_ANY);
}

// tracepoint：tcp_connect → 建立 {fd, pid → 五元组} 映射  
SEC("tracepoint/sock/inet_sock_set_state")
int trace_tcp_connect(struct trace_event_raw_inet_sock_set_state *ctx) {
    // 提取 saddr, daddr, sport, dport
    // 写入 fd_to_tuple map
}

// uprobe 2：HKDF → 读两张 map，输出完整记录
SEC("uprobe/.../chrome:HKDF_RVA")
int hook_hkdf(struct pt_regs *ctx) {
    // 1. 读 ssl_ptr → fd
    // 2. 读 fd → 五元组
    // 3. 读 HKDF 参数 → 密钥
    // 4. 合并输出到 ring_buffer
}
```

这种方式比 Frida 方案在时序上更准确——所有关联发生在内核同一上下文中，不存在用户态轮询 `/proc/net/tcp` 时连接已消失的竞态问题。

## 四、方案综合设计建议

综合三个问题，推荐的实现架构如下：

**离线部分（数据库构建）**
对每个浏览器版本，TLSKeyHunter 自动提取 `PRF RVA`、`HKDF RVA`，额外通过符号表自动获取 `SSL_set_fd RVA`，通过对比源码自动提取 `ssl_st.fd 偏移量`。Chrome/Edge/Brave 共享同一套 BoringSSL 指纹，Firefox 单独维护 NSS 指纹库。

**在线部分（运行时提取，Linux）**
优先使用 eBPF uprobe 方案：加载三个探针（`SSL_set_fd`、`HKDF`、`tcp_connect tracepoint`），通过三张 BPF Map 关联，输出携带五元组的完整 Key Log。对于需要支持 macOS/Windows 的场景，降级使用 Frida 方案，逻辑相同但通过 Frida 的 `Memory.readByteArray` 和 `Socket.peerAddress` 替代 eBPF 的内存读取。

**回退机制**
版本数据库未命中时，先用大版本指纹在 `/proc/PID/maps` 定位的模块范围内进行字节序列扫描，动态计算 RVA，再挂载探针。若扫描也失败，则触发一次轻量级静态分析（Ghidra headless 模式），将结果写回数据库供后续使用。

## 重新给你的指纹提取脚本（Java，完整可用版）

这个版本直接对照 TLSKeyHunter 的核心逻辑重写，确保与其输出格式一致：

```java
// ExtractKDFFingerprint.java
// 使用方法：光标放在目标函数（FUN_0a32d4b0）内任意位置，运行脚本
// 输出：函数入口地址(RVA) + 指纹字节序列（与 TLSKeyHunter 格式兼容）

import ghidra.app.script.GhidraScript;
import ghidra.program.model.address.Address;
import ghidra.program.model.listing.Function;
import ghidra.program.model.listing.Instruction;
import ghidra.program.model.listing.InstructionIterator;

public class ExtractKDFFingerprint extends GhidraScript {

    // 最小指纹长度（字节），与 TLSKeyHunter 论文一致
    private static final int MIN_FINGERPRINT_BYTES = 32;

    @Override
    public void run() throws Exception {

        // 1. 获取当前光标所在函数
        Function func = getFunctionContaining(currentAddress);
        if (func == null) {
            printerr("[-] 未找到函数，请将光标放在目标函数内部后重新运行");
            return;
        }

        Address entryPoint = func.getEntryPoint();
        println("[*] 函数名称 : " + func.getName());
        println("[*] 函数入口 : " + entryPoint);

        // 2. 计算 RVA（减去程序镜像基址）
        long imageBase = currentProgram.getImageBase().getOffset();
        long rva = entryPoint.getOffset() - imageBase;
        println("[*] RVA (基址已减) : 0x" + Long.toHexString(rva).toUpperCase());

        // 3. 从函数入口开始遍历指令，提取字节指纹
        InstructionIterator instrIter = currentProgram.getListing()
                .getInstructions(entryPoint, true);

        StringBuilder hexBuilder = new StringBuilder();
        int totalBytes = 0;
        boolean done = false;

        while (instrIter.hasNext() && !done) {
            Instruction instr = instrIter.next();

            // 确保仍在本函数体内
            if (!func.getBody().contains(instr.getAddress())) {
                break;
            }

            String mnemonic = instr.getMnemonicString().toUpperCase();
            byte[] raw = instr.getBytes();

            // 收集该指令的字节
            for (byte b : raw) {
                hexBuilder.append(String.format("%02X ", b));
            }
            totalBytes += raw.length;

            // 判断停止条件：
            // 遇到任何跳转/返回指令，且已经满足最小长度 → 停止
            // CALL 不停止（允许穿越 CALL 继续收集）
            boolean isJump = mnemonic.startsWith("J")     // JZ, JNZ, JMP, JG 等
                          || mnemonic.equals("RET")
                          || mnemonic.equals("RETN")
                          || mnemonic.equals("RETF");

            // JMP 是无条件跳转，遇到后必须停止（即使不足32字节）
            boolean isUnconditionalJump = mnemonic.equals("JMP");

            if (isUnconditionalJump) {
                done = true;
            } else if (isJump && totalBytes >= MIN_FINGERPRINT_BYTES) {
                done = true;
            }
        }

        // 4. 输出结果
        println("");
        println("===== 指纹提取结果 =====");
        println("[+] 函数名称    : " + func.getName());
        println("[+] 入口地址    : 0x" + entryPoint.toString().toUpperCase());
        println("[+] RVA         : 0x" + Long.toHexString(rva).toUpperCase());
        println("[+] 指纹长度    : " + totalBytes + " 字节");
        println("[+] 指纹字节序列 (Frida 格式):");
        println("    " + hexBuilder.toString().trim());
        println("");
        println("===== 写入数据库的 JSON 片段 =====");
        println("{");
```

## 三、⚠️ RVA 计算需要确认一个问题
