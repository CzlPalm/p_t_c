# T6.1：ssl_log_secret 参数探针验证

**阶段**: P6  
**任务**: T6.1 ssl_log_secret 参数探针验证  
**状态**: 已完成代码准备，待运行验证

---

## 一、任务目标

根据 `learn/P6/P6_execution_plan_v2.md`，T6.1 的目标是：

1. 验证 `ssl_log_secret()` 的参数顺序
2. 验证 `args[0]` 是否是真正可用的 `ssl_st*`
3. 验证是否可以通过 `ssl_st -> rbio -> fd` 直接恢复有效 fd
4. 为后续 T6.2 正式集成 `ssl_log_secret` Hook 提供参数依据

## 五、预期验证点

如果参数布局符合预期，通常应观察到近似形式：

- `args[0] = ssl_st*`
- `args[1] = label ptr`
- `args[2] = secret ptr`
- `args[3] = secret len`

并希望出现：

- `label = "CLIENT_RANDOM" / "CLIENT_HANDSHAKE_TRAFFIC_SECRET" / ...`
- `fd > 0`
- `client_random != null`

---

## 六、关键判定标准

### 判定 A：参数顺序是否成立

若日志中：

- `args[1]` 能稳定解出 label 字符串
- `args[3]` 能稳定表现为合理长度（32 / 48 等）
- `args[2]` 能按该长度正确读出 secret

则可判定参数顺序基本成立。

### 判定 B：args[0] 是否是真正的 ssl_st*

若：

- `ssl + 0x240 -> rbio`
- `rbio + 0x03c -> fd`

可以稳定读出有效 `fd (> 0)`，则说明 `args[0]` 很可能是真实 `ssl_st*`。

若持续出现：

- `rbio = NULL`
- `fd = -1`
- access violation

则说明 `args[0]` 仍可能不是可直接按 `ssl_st` 解释的对象，或该函数所在调用上下文的结构布局与 HKDF 不同，需要进一步验证。

---

## 七、与 P5 已知问题的关系

本任务直接针对 P5 两个关键遗留问题：

1. **TLS 1.2 PRF 复用/恢复导致 `CLIENT_RANDOM` 漏捕**
   - 若 `ssl_log_secret()` 能输出这些条目，将成为补漏关键路径

2. **fd 精准匹配失败**
   - 若 `ssl_log_secret()` 的 `args[0]` 真正接近 `ssl_st*`，则有机会恢复 fd 精准五元组关联

因此，T6.1 是 P6 的关键起始任务。
