## 一、P6 阶段目标

P6 的目标不再是继续打磨单版本 Chrome 工具，而是正式进入你项目的长期主目标：

> **针对 Chrome 多个历史版本与流行版本，自动采集二进制、批量分析 TLS 关键函数，提取指纹/特征码/RVA，并构建可持续维护的浏览器密钥指纹数据库。**

同时，P6 还将引入第二条关键增强主线：

> **集成 `ssl_log_secret()` 作为 Chrome 专用补充 Hook 点，以提高密钥覆盖率，并弥补当前 PRF 复用导致的 TLS 1.2 `CLIENT_RANDOM` 漏捕问题。**

因此，P6 的本质是“双主线并行”：

1. **数据库化主线**：自动采集 + 批量分析 + 版本记录入库
2. **覆盖率增强主线**：`ssl_log_secret()` 集成与当前 PRF/HKDF 体系互补

## 二、P6 阶段核心问题定义

P5 已明确暴露出两个关键限制：

### 问题 1：版本扩展能力仍然缺失

当前项目只有一条实际可用记录：

- `chrome_143.0.7499.169_linux_x86_64.json`

这意味着当前工具仍然是：

- 单浏览器
- 单版本
- 人工填表式配置

距离“数据库化、多版本复用、自动化扩展”还有明显距离。

### 问题 2：TLS 1.2 复用路径存在漏捕

P5 测试中已确认：

- `hook_total = 419`
- `env_total = 498`
- 缺失 79 条，且全部是 `CLIENT_RANDOM`

这说明：

- PRF/HKDF Hook 准确率高
- 但对 TLS 1.2 复用/恢复场景召回率不足

而 `ssl_log_secret()` 正是当前最有希望补足这一缺口的 Chrome 专用路径。

## 四、P6 设计原则

### 4.1 版本数据库优先于代码分支

P6 不应通过在 `tls_capture.py` 中不断加 `if version == xxx` 的方式扩展，而应始终坚持：

- 版本差异进入 JSON / 数据库
- Hook 脚本模板保持稳定
- 分析流水线自动生成记录

### 4.2 `ssl_log_secret()` 作为 Chrome 专用优先路径

在确认目标浏览器是 Chrome/BoringSSL 且该版本已有 `ssl_log_secret()` 记录时：

- 优先尝试该函数 Hook
- 成功则直接使用其输出
- 失败再回退 PRF/HKDF

### 4.3 PRF/HKDF 保留为基础能力与回退路径

因为：

- 并非所有版本都能保证 `ssl_log_secret()` 易于稳定 Hook
- 非 Chrome 浏览器未来也未必存在这个函数
- PRF/HKDF 是跨实现的基础锚点

因此，P6 的策略不是替换，而是互补。

### 4.4 所有新版本记录必须带验证状态

数据库不是“分析结果堆积”，而是“经过验证的可用知识库”。

每条记录至少应标记：

- `located`：仅定位成功
- `hooked`：Hook 成功
- `diff_verified`：与 env 对比通过
- `wireshark_verified`：可解密验证通过

# 主线 A：Chrome 多版本二进制采集与自动化批量分析

## P6.A.1 版本采集策略设计

### 目标

建立 Chrome Linux 版本样本池，覆盖：

- 若干近版本
- 若干历史主版本
- 若干小版本补丁差异样本

### 建议优先覆盖区间

第一批建议先覆盖：

- 当前版本所在大版本前后若干个主版本
- 每个主版本选 1~3 个代表性补丁版本

例如：

- 140.x
- 141.x
- 142.x
- 143.x
- 144.x

### 采集内容

每个版本至少保存：

- 浏览器版本号
- 下载 URL
- 二进制文件路径
- SHA256
- 平台/架构
- 获取时间

### 产出物

建议新增目录：

```text
artifacts/
  chrome/
    143.0.7499.169/
      chrome
      metadata.json
    143.0.7499.119/
      chrome
      metadata.json
```

## P6.A.2 自动下载器实现

### 目标

实现可批量抓取 Chrome 指定版本 Linux 二进制的下载脚本。

### 推荐能力

下载器至少应支持：

- 输入版本列表批量下载
- 自动校验文件存在性
- 自动记录 metadata
- 失败重试
- 跳过已下载版本

### 注意事项

需要先确认：

- Chrome 官方是否直接提供稳定历史 Linux 包索引
- 若无统一索引，是否需要从公开仓库/镜像源/包仓元数据推导
- 若 Chrome 官方历史版本不便获取，可先用 Chromium / apt 仓库快照过渡

**结论**：P6 初期可以先解决“可获取一批可分析版本”，不必一开始就追求全网覆盖完整历史库。

## P6.A.3 批量静态分析流水线

### 目标

对每个版本自动提取：

- PRF RVA / 指纹
- HKDF RVA / 指纹
- 可选的 `ssl_log_secret()` RVA / 指纹
- 关键偏移 / 标签映射
- 版本记录 JSON

### 分析来源

P6 推荐采用“三源融合”：

1. **TLSKeyHunter**
   - 自动提取 KDF 指纹
2. **BoringSecretHunter**
   - 自动定位 `ssl_log_secret()`
3. **自定义补充分析脚本**
   - 弥补 TLSKeyHunter / BSH 输出字段不足
   - 补齐 JSON 模板需要的信息

### 输出形态

每个版本产出一份结构化记录：

```json
{
  "meta": { ... },
  "hook_points": {
    "prf": { ... },
    "hkdf": { ... },
    "ssl_log_secret": { ... }
  },
  "tls13_label_map": { ... },
  "struct_offsets": { ... },
  "validation": { ... }
}
```

## P6.A.4 指纹数据库模型设计

### 目标

把当前单个 JSON 配置，提升为真正的“版本指纹数据库记录模型”。

### 每条记录至少应包含

- 浏览器家族：`chrome`
- 平台：`linux`
- 架构：`x86_64`
- 精确版本号
- 二进制 hash
- TLS 库类型
- Hook 点信息：
  - `rva`
  - `fingerprint`
  - `fingerprint_len`
  - `role`
  - `source`
  - `status`
- `client_random` 路径
- TLS 1.3 label 映射
- 结构偏移
- 五元组建议策略
- 验证状态
- 分析来源
- 更新时间

### 阶段建议

P6 先继续使用“每版本一份 JSON”即可，暂不急着上数据库服务。

因为：

- 当前项目还处在研究工程阶段
- 文件式 JSON 更易 diff、回滚、人工修订
- 真正的数据库服务可以留到后续阶段

# 主线 B：集成 `ssl_log_secret()` 提升 Chrome 覆盖率

## P6.B.1 目标与价值

当前已知：

- 通过 BoringSecretHunter 已经定位到当前 Chrome 版本的 `ssl_log_secret()`
- P5 漏掉的 79 条全是 TLS 1.2 `CLIENT_RANDOM`
- 这些缺失高度怀疑属于 PRF 复用/恢复类路径

因此，P6 要验证的核心假设是：

> **如果 Hook `ssl_log_secret()`，就有机会获得浏览器最终输出的 keylog 条目，从而补足 PRF/HKDF 在 TLS 1.2 恢复路径下的漏捕。**

---

## P6.B.2 集成策略

### Chrome 目标下的运行时决策建议

当 `browser == chrome` 且版本配置中存在 `ssl_log_secret` 时：

1. 优先启用 `ssl_log_secret()` Hook
2. 同时保留 PRF/HKDF Hook
3. 按去重规则合并输出
4. 如果 `ssl_log_secret()` Hook 初始化失败：
   - 回退到当前 PRF/HKDF 模式

### 为什么建议“双开而不是替代”

因为当前我们还没有证据证明：

- `ssl_log_secret()` 能覆盖全部条目
- 它一定能稳定拿到五元组所需上下文
- 它在所有版本上都足够稳

所以 P6 最合理的策略不是替换，而是：

> **让 `ssl_log_secret()` 做增强源，让 PRF/HKDF 做基础源。**

---

## P6.B.3 需要验证的关键问题

### 问题 1：`ssl_log_secret()` 是否能补足 79 条缺失的 `CLIENT_RANDOM`

这是 P6.B 的头号验收点。

### 问题 2：`ssl_log_secret()` 的参数里是否更接近真正的 `ssl_st*`

如果是，那么它可能同时帮助：

- 覆盖 TLS 1.2 复用漏捕
- 改善 fd 精准关联可能性

### 问题 3：与当前输出如何去重

需要确认是否会出现：

- 同一条 secret 既被 `ssl_log_secret()` 输出，又被 HKDF/PRF 输出

因此必须制定统一去重键。

建议仍以：

- `label + client_random`

作为首层去重键，必要时再补 `secret` 比较。

---

## P6.B.4 五元组关联预期

### 最乐观情况

如果 `ssl_log_secret()` 的上下文里可以恢复出更真实的 `ssl_st*` 或可追溯 fd，那么：

- 有机会重新启用 fd 精准关联探索
- 可使 Chrome 路径获得比 HKDF 更好的连接归属能力

### 保守情况

如果仍无法稳定提取 fd，那么：

- 继续使用当前时序关联 + cache
- 只把 `ssl_log_secret()` 作为覆盖率增强源

### 阶段策略

P6 不把“恢复 fd 精准匹配”作为必须达成目标；
P6 只要求优先验证：

1. 覆盖率是否提升
2. 与现有输出是否一致
3. 五元组关联是否不退化

## 八、P6 风险与应对

### 风险 1：历史 Chrome 二进制获取困难

**应对**：

- 先优先获取能拿到的一批版本
- 必要时先以 Chromium / apt snapshot 替代部分样本
- 不因“历史全覆盖”阻塞 P6 启动

### 风险 2：TLSKeyHunter / BoringSecretHunter 批处理速度慢

**应对**：

- 分层缓存中间产物
- 分离“下载”“分析”“汇总”三个阶段
- 先小批量试跑再扩大规模

### 风险 3：`ssl_log_secret()` 不易稳定 Hook

**应对**：

- 保留 PRF/HKDF 基础链路
- `ssl_log_secret()` 作为增强源而非唯一依赖

### 风险 4：`ssl_log_secret()` 不一定直接改善五元组关联

**应对**：

- 将“覆盖率提升”设为主目标
- 将“fd 恢复”设为 bonus 目标
