原始 `watchdog_attach_v2.py` 中 `readCR` 函数：

```
// 即：s3 = *ssl_ptr → sub = *(s3+0x30) → cr = sub+0x30（直接读，不再解引用）
```

## diff 结果解读

**好消息**：diff 里**只有 `>` 行，没有 `<` 行**。这意味着 Frida 捕获的 125 条密钥**全部正确**（零误报），P3 的正确性验证通过了。

**剩余问题**：有约 50 条 SSLKEYLOGFILE 里的记录 Frida 没有捕获到，仔细看这些缺失项分为两类：

**类型一：48字节密钥被截断（最多缺失，根本原因）**

```
> CLIENT_HANDSHAKE_TRAFFIC_SECRET 0659befa... a09966c7...d5423a   ← 96个十六进制字符 = 48字节
> CLIENT_HANDSHAKE_TRAFFIC_SECRET 0c15d0b3... 3e5685ec...4f7      ← 48字节
> SERVER_TRAFFIC_SECRET_0 947793b3... 9cd2182d...f45b8           ← 48字节
```

这些连接用的是 `TLS_AES_256_GCM_SHA384`，密钥长度是48字节。当前脚本在 `onLeave` 里读 `ssl_ptr+0xb2` 这个长度字节，但该字节是在 HKDF 函数**返回之后才被写入**的，所以 `onLeave` 读到的是0，回退到默认值32，导致密钥被截断，与 env log 不匹配而被丢弃。

**类型二：极少量早期连接丢失（时序问题）**

```
> CLIENT_HANDSHAKE_TRAFFIC_SECRET 155d7d3c... 60e73572...  ← 32字节，也缺失
```

这是 Chrome 启动时 Frida 还没附加就已经完成握手的极少数连接，无法完全避免。

## 根本修复：从 EVP_MD 结构体读取哈希长度

`FUN_049837e0` 的 `args[5]`（R9 = param_14）是 `EVP_MD*` 指针，BoringSSL 的 `EVP_MD` 结构体在偏移 `+4` 处存放 `md_size`（int，4字节），直接读它就能得到准确的密钥长度（SHA-256 → 32，SHA-384 → 48），完全不依赖 ssl_ptr 偏移。

修改非常小，只需改 HOOK_JS 里的 HKDF hook 部分：

```javascript
HOOK_JS = r"""
'use strict';

const NSS_LABEL = {
    'c e traffic':  'CLIENT_EARLY_TRAFFIC_SECRET',
    'c hs traffic': 'CLIENT_HANDSHAKE_TRAFFIC_SECRET',
    's hs traffic': 'SERVER_HANDSHAKE_TRAFFIC_SECRET',
    'c ap traffic': 'CLIENT_TRAFFIC_SECRET_0',
    's ap traffic': 'SERVER_TRAFFIC_SECRET_0',
    'exp master':   'EXPORTER_SECRET',
};

function hex(buf) {
    if (!buf) return null;
    return Array.from(new Uint8Array(buf))
        .map(b => b.toString(16).padStart(2, '0')).join('');
}

function readCR(ssl_ptr) {
    // 已验证正确路径: *(*(*(ssl_ptr) + 0x30) + 0x30) → 32字节
    try {
        const s3  = ssl_ptr.readPointer();
        const sub = s3.add(0x30).readPointer();
        return sub.add(0x30).readByteArray(32);
    } catch(_) { return null; }
}

function emit(line) { send({t: 'key', v: line}); }
function dbg(msg)   { send({t: 'dbg', v: '[pid=' + Process.id + '] ' + msg}); }

const mod = (function() {
    for (const n of ['chrome', 'libchrome.so']) {
        try { return Process.getModuleByName(n); } catch(_) {}
    }
    return Process.enumerateModules()
        .filter(m => m.size > 50 * 1024 * 1024)
        .sort((a, b) => b.size - a.size)[0] || null;
})();

if (!mod) {
    dbg('chrome 模块未找到');
    send({t: 'no_module'});
} else {
    dbg('模块: ' + mod.name + ' base=' + mod.base +
        ' size=' + (mod.size / 1024 / 1024 | 0) + 'MB');
    let prf_ok = false, hkdf_ok = false;

    // ── TLS 1.2 PRF Hook ──
    try {
        Interceptor.attach(mod.base.add(ptr('0x0A22D4B0')), {
            onEnter(args) {
                this.ssl_ptr = args[0];
                this.out_ptr = args[1];
            },
            onLeave(_) {
                try {
                    const cr = readCR(this.ssl_ptr);
                    const ms = this.out_ptr.readByteArray(48);
                    if (cr && ms) emit('CLIENT_RANDOM ' + hex(cr) + ' ' + hex(ms));
                } catch(_) {}
            }
        });
        prf_ok = true;
        dbg('PRF hook OK @ ' + mod.base.add(ptr('0x0A22D4B0')));
    } catch(e) { dbg('PRF 失败: ' + e); }

    // ── TLS 1.3 HKDF Hook ──
    try {
        Interceptor.attach(mod.base.add(ptr('0x048837E0')), {
            onEnter(args) {
                this.ssl_ptr = args[0];
                this.out_ptr = args[1];
                // args[5] = R9 = EVP_MD*  →  偏移+4 = md_size (int)
                this.md_ptr  = args[5];
                const ll = args[4].toInt32();
                try {
                    this.lbl = (ll > 0 && ll <= 20)
                        ? args[3].readUtf8String(ll) : null;
                } catch(_) { this.lbl = null; }
            },
            onLeave(_) {
                try {
                    if (!this.lbl) return;
                    const nss = NSS_LABEL[this.lbl];
                    if (!nss) return;

                    // 从 EVP_MD.md_size（偏移+4）读取哈希长度
                    // SHA-256 → 32字节，SHA-384 → 48字节
                    let kl = 32;
                    try {
                        const md_size = this.md_ptr.add(4).readU32();
                        if (md_size === 32 || md_size === 48) kl = md_size;
                    } catch(_) {}

                    const secret = this.out_ptr.readByteArray(kl);
                    const cr     = readCR(this.ssl_ptr);
                    if (secret && cr) emit(nss + ' ' + hex(cr) + ' ' + hex(secret));
                } catch(_) {}
            }
        });
        hkdf_ok = true;
        dbg('HKDF hook OK @ ' + mod.base.add(ptr('0x048837E0')));
    } catch(e) { dbg('HKDF 失败: ' + e); }

    send({t: 'ready', prf: prf_ok, hkdf: hkdf_ok});
}
"""
```

## 三、根本原因分析

### 原因一（最可能）：Chrome 多进程，只 Hook 了一个 NetworkService

Chrome 在 Linux 上会有多个进程，执行 `pgrep -a chrome | grep NetworkService` 可能返回不止一个 PID：

```bash
pgrep -a chrome | grep NetworkService
# 可能有：
# 52234 ... --type=utility --utility-sub-type=network.mojom.NetworkService ...
# 52890 ... --type=utility --utility-sub-type=network.mojom.NetworkService ...
```

`watchdog_attach_v2.py` 的逻辑是找到**第一个** NetworkService 进程就附加，如果 Chrome 有两个网络进程（比如隐身窗口 + 普通窗口，或重启后遗留旧进程），漏掉的 session 就来自另一个进程。

**验证方法：**
```bash

```

### 原因二（次可能）：时序问题，Hook 注册前已建立连接

SSLKEYLOGFILE 在 Chrome **启动时就开始写入**，但 Frida 要等 Python 脚本找到进程、`attach`、`load script` 完成之后才开始 Hook，这中间有 2-5 秒窗口，期间 Chrome 建立的连接全部漏掉。

漏掉的条目如果集中在 env.log 的**前段**，就是这个原因。

**验证方法：**

```python
# 找到所有网络进程

    descendants = get_all_descendants(parent_pid)
    net_pids = []
    for pid in descendants:
```

## 修复方案：watchdog_attach_v4.py（spawn 模式）

```python
#!/usr/bin/env python3
"""
watchdog_attach_v4.py
─────────────────────────────────────────────────────────────
使用 Frida spawn 模式：在 Chrome 第一条指令执行前注入，
确保零时序差，不漏任何 TLS 连接。

用法：
    sudo python3 watchdog_attach_v4.py
"""

import frida, sys, os, time, signal, threading

OUTPUT_FILE  = '/tmp/chrome_sslkeys_frida_v4.log'
CHROME_BIN   = '/opt/google/chrome/chrome'
USER_DATA    = '/tmp/chrome_p3_test'
ENV_LOG      = '/tmp/chrome_sslkeys_env.log'

# ── Hook 脚本（与 v3 相同，仅 client_random 路径已修正为 +0x30）──
HOOK_JS = r"""
'use strict';

const NSS_LABEL = {
    'c e traffic':  'CLIENT_EARLY_TRAFFIC_SECRET',
    'c hs traffic': 'CLIENT_HANDSHAKE_TRAFFIC_SECRET',
    's hs traffic': 'SERVER_HANDSHAKE_TRAFFIC_SECRET',
    'c ap traffic': 'CLIENT_TRAFFIC_SECRET_0',
    's ap traffic': 'SERVER_TRAFFIC_SECRET_0',
    'exp master':   'EXPORTER_SECRET',
};

// 密钥长度字节在 ssl_ptr 上的偏移
const KEY_LEN_OFFSET = {
    'c e traffic':  0x81,
    'c hs traffic': 0xb2,
    's hs traffic': 0xe3,
    'c ap traffic': 0x114,
    's ap traffic': 0x145,
};

function hex(buf) {
    if (!buf) return null;
    return Array.from(new Uint8Array(buf))
        .map(b => b.toString(16).padStart(2, '0')).join('');
}

// 已确认正确路径：*(*(*(ssl_ptr) + 0x30) + 0x30) → 32字节
function readCR(ssl) {
    try {
        const s3  = ssl.readPointer();           // *ssl_ptr = s3
        const sub = s3.add(0x30).readPointer();  // *(s3 + 0x30)
        return sub.add(0x30).readByteArray(32);  // client_random ← +0x30（已验证）
    } catch(_) { return null; }
}

function emit(line) { send({t: 'key', v: line}); }
function dbg(msg)   { send({t: 'dbg', v: '[pid=' + Process.id + '] ' + msg}); }

// ── 等待 chrome 模块加载 ──
function waitForModule(name, cb, retry) {
    retry = retry || 0;
    try {
        const m = Process.getModuleByName(name);
        cb(m);
    } catch(_) {
        if (retry > 100) { dbg('模块等待超时: ' + name); return; }
        setTimeout(() => waitForModule(name, cb, retry + 1), 100);
    }
}

function installHooks(mod) {
    dbg('模块: ' + mod.name + ' base=' + mod.base +
        ' size=' + (mod.size / 1024 / 1024 | 0) + 'MB');

    let prf_ok = false, hkdf_ok = false;

    // ── TLS 1.2 PRF ──────────────────────────────────────────
    try {
        Interceptor.attach(mod.base.add(ptr('0x0A22D4B0')), {
            onEnter(args) {
                // args[0]=RDI=ssl_ptr, args[1]=RSI=output(master_secret)
                this.ssl = args[0];
                this.out = args[1];
            },
            onLeave(_) {
                try {
                    const cr = readCR(this.ssl);
                    const ms = this.out.readByteArray(48);
                    if (cr && ms) emit('CLIENT_RANDOM ' + hex(cr) + ' ' + hex(ms));
                } catch(_) {}
            }
        });
        prf_ok = true;
        dbg('PRF hook OK @ ' + mod.base.add(ptr('0x0A22D4B0')));
    } catch(e) { dbg('PRF 失败: ' + e); }

    // ── TLS 1.3 HKDF ─────────────────────────────────────────
    try {
        Interceptor.attach(mod.base.add(ptr('0x048837E0')), {
            onEnter(args) {
                // args[0]=RDI=ssl_ptr, args[1]=RSI=output_buf
                // args[3]=RCX=label_ptr, args[4]=R8=label_len
                this.ssl = args[0];
                this.out = args[1];
                try {
                    const ll = args[4].toInt32();
                    this.lbl = (ll > 0 && ll <= 20)
                        ? args[3].readUtf8String(ll)
                        : null;
                } catch(_) { this.lbl = null; }
            },
            onLeave(_) {
                try {
                    if (!this.lbl) return;
                    const nss = NSS_LABEL[this.lbl];
                    if (!nss) return;

                    // 动态读取密钥长度（支持 SHA-256=32 和 SHA-384=48）
                    const lo = KEY_LEN_OFFSET[this.lbl];
                    let kl = lo ? this.ssl.add(lo).readU8() : 32;
                    if (kl === 0 || kl > 64) kl = 32;

                    const secret = this.out.readByteArray(kl);
                    const cr     = readCR(this.ssl);
                    if (secret && cr)
                        emit(nss + ' ' + hex(cr) + ' ' + hex(secret));
                } catch(_) {}
            }
        });
        hkdf_ok = true;
        dbg('HKDF hook OK @ ' + mod.base.add(ptr('0x048837E0')));
    } catch(e) { dbg('HKDF 失败: ' + e); }

    send({t: 'ready', prf: prf_ok, hkdf: hkdf_ok});
}

// spawn 模式下，chrome 模块可能还未加载，需要等待
waitForModule('chrome', installHooks);
"""

# ─────────────────────────────────────────────
device   = frida.get_local_device()
sessions = {}   # pid → session（支持多 session）
keycount = 0
lock     = threading.Lock()

def on_message(pid):
    def handler(message, _):
        global keycount
        if message.get('type') != 'send': return
        p = message.get('payload', {})
        t = p.get('t', '')
        if t == 'key':
            with lock:
                keycount += 1
                line = p['v']
            print(f'\033[32m[KEY #{keycount} pid={pid}]\033[0m {line}')
            with open(OUTPUT_FILE, 'a') as f:
                f.write(line + '\n')
        elif t == 'dbg':
            print(f'\033[36m[DBG pid={pid}]\033[0m {p["v"]}')
        elif t == 'ready':
            prf  = '✓' if p.get('prf')  else '✗'
            hkdf = '✓' if p.get('hkdf') else '✗'
            print(f'\033[32m[+]\033[0m pid={pid}  PRF={prf}  HKDF={hkdf}')
        elif t == 'no_module':
            print(f'\033[31m[-]\033[0m pid={pid} chrome 模块未找到')
        elif message.get('type') == 'error':
            print(f'\033[31m[ERR pid={pid}]\033[0m {message.get("stack","")}')
    return handler

def attach_pid(pid, label=''):
    try:
        sess = device.attach(pid)
        scr  = sess.create_script(HOOK_JS)
        scr.on('message', on_message(pid))
        scr.load()
        sessions[pid] = sess
        print(f'\033[32m[+]\033[0m 附加成功 pid={pid} {label}')
        return True
    except Exception as e:
        print(f'\033[31m[-]\033[0m 附加失败 pid={pid}: {e}')
        return False

# ── 监控子进程，自动附加新建的 NetworkService ──
def watch_child_processes(parent_pid):
    known = set()
    while True:
        try:
            for entry in os.scandir('/proc'):
                if not entry.name.isdigit(): continue
                pid = int(entry.name)
                if pid in known or pid in sessions: continue
                try:
                    with open(f'/proc/{pid}/status') as f:
                        for line in f:
                            if line.startswith('PPid:'):
                                ppid = int(line.split()[1])
                                break
                    if ppid != parent_pid: continue
                    with open(f'/proc/{pid}/cmdline', 'rb') as f:
                        cmd = f.read().decode('utf-8', errors='ignore')
                    if 'NetworkService' in cmd:
                        known.add(pid)
                        time.sleep(0.3)  # 等进程稳定
                        attach_pid(pid, '[NetworkService]')
                    else:
                        known.add(pid)
                except: pass
        except: pass
        time.sleep(0.2)

def cleanup(sig=None, frame=None):
    print()
    for pid, sess in sessions.items():
        try: sess.detach()
        except: pass
    print(f'[*] 共捕获 {keycount} 条密钥')
    print(f'[*] 验证命令：')
    print(f'    diff <(sort {OUTPUT_FILE}) \\')
    print(f'         <(grep -E "^(CLIENT|SERVER|EXPORTER)" {ENV_LOG} | sort)')
    sys.exit(0)

signal.signal(signal.SIGINT, cleanup)

# ─────────────────────────────────────────────
print('=' * 55)
print('  Chrome TLS Watchdog v4（spawn 模式，零时序差）')
print('=' * 55)

# 清空日志
open(OUTPUT_FILE, 'w').close()
open(ENV_LOG,     'w').close()

# spawn Chrome
print(f'[*] spawn: {CHROME_BIN}')
env = os.environ.copy()
env['SSLKEYLOGFILE'] = ENV_LOG

child_pid = device.spawn(
    [CHROME_BIN,
     '--no-sandbox',
     f'--user-data-dir={USER_DATA}',
     '--disable-extensions'],
    env=env
)
print(f'[*] Chrome PID = {child_pid}')

# 注入主进程（spawn 暂停状态下先注入）
attach_pid(child_pid, '[main]')

# 启动子进程监控线程（自动 attach 新的 NetworkService）
t = threading.Thread(target=watch_child_processes,
                     args=(child_pid,), daemon=True)
t.start()

# 恢复 Chrome 执行
device.resume(child_pid)
print(f'\033[32m[+]\033[0m Chrome 已恢复运行')
print(f'[+] 访问 HTTPS 站点后 Ctrl+C 退出\n')

try:
    while True:
        time.sleep(1)
except KeyboardInterrupt:
    pass

cleanup()
```

### 问题 A：EXPORTER_SECRET 密钥截断（最好修）

对比一条：
```
Frida:  548fdc...474a8                      ← 32字节（64个hex字符）
Env:    548fdc...474a8 13aa8c...eb9         ← 48字节（96个hex字符）
```

Frida 捕获了前32字节，剩余16字节截断丢失。原因明确：`KEY_LEN_OFFSET` 表里**没有 `'exp master'`**，导致长度 fallback 到32，但这些连接使用 SHA-384 套件，实际密钥长48字节。

**修复**：在 Ghidra 里找 `exp master` 对应的长度偏移字节，补入表中。根据已有偏移的规律：

```
c hs → 密钥起始 0x82，长度字节 0xb2（差0x30）
s hs → 密钥起始 0xb3，长度字节 0xe3（差0x30）
c ap → 密钥起始 0xe4，长度字节 0x114（差0x30）
s ap → 密钥起始 0x115，长度字节 0x145（差0x30）
```

规律非常清晰，每个密钥的长度字节在密钥起始偏移 `+0x30` 处。`exp master` 的存储偏移是 `*(ssl_ptr[0]+0x30)+0x182`，是二级解引用，偏移表需要特殊处理。

临时修复方案——在 onLeave 里动态读取实际长度：

```javascript
// 在 HKDF hook 的 onLeave 里，exp master 特殊处理
if (this.lbl === 'exp master') {
    try {
        const s3  = this.ssl.readPointer();
        const sub = s3.add(0x30).readPointer();
        kl = sub.add(0x1b2).readU8();  // exp master 的长度字节偏移
        if (kl === 0 || kl > 64) kl = 32;
    } catch(_) { kl = 32; }
}
```

# 一、exp master 长度修复

从反编译代码可以直接读出完整路径：

```c
// lVar2 = *param_9 = *ssl_ptr (= s3 指针)
// exp master 输出位置: *(lVar2 + 0x30) + 0x182
// 长度字节位置:        *(byte*)(*(lVar2 + 0x30) + 0x1b2)
```

这是**二级解引用路径**，与其他密钥（直接 `ssl_ptr + offset`）不同，所以 `KEY_LEN_OFFSET` 表无法直接覆盖。修复就是在 `onLeave` 里对 `exp master` 单独处理：

```javascript
onLeave(_) {
    try {
        if (!this.lbl) return;
        const nss = NSS_LABEL[this.lbl];
        if (!nss) return;

        let kl;
        if (this.lbl === 'exp master') {
            // 二级解引用路径：*(*(*(ssl_ptr)) + 0x30) + 0x1b2
            try {
                const s3  = this.ssl.readPointer();        // *ssl_ptr
                const sub = s3.add(0x30).readPointer();    // *(s3 + 0x30)
                kl = sub.add(0x1b2).readU8();              // 长度字节
                if (kl === 0 || kl > 64) kl = 32;
            } catch(_) { kl = 48; }  // SHA-384 时 fallback 到48
        } else {
            const lo = KEY_LEN_OFFSET[this.lbl];
            kl = lo ? this.ssl.add(lo).readU8() : 32;
            if (kl === 0 || kl > 64) kl = 32;
        }

        const secret = this.out.readByteArray(kl);
        const cr     = readCR(this.ssl);
        if (secret && cr)
            emit(nss + ' ' + hex(cr) + ' ' + hex(secret));
    } catch(_) {}
}
```

## 三、BoringSecretHunter 融合方案

你的提议非常有价值，这个融合在技术上完全可行，而且能大幅简化数据库方案。

### BoringSecretHunter 的本质

它 hook 的是 BoringSSL 内部的 `ssl_log_secret()` 函数。这个函数在 BoringSSL 源码中的签名是：

```c
void ssl_log_secret(const SSL *ssl,        // ssl 对象
                    const char *label,     // NSS 格式标签（"CLIENT_HANDSHAKE_TRAFFIC_SECRET"等）
                    const uint8_t *secret, // 密钥字节
                    size_t secret_len);    // 密钥长度
```

这正是 SSLKEYLOGFILE 机制的**内部实现函数**——环境变量方式和 hook 这个函数走的是同一条代码路径，所以输出必然完全一致。

### 两种 Hook 方案的对比

| 维度 | 当前方案（HKDF级） | BoringSecretHunter方案（ssl_log_secret级） |
|------|------|------|
| Hook 层次 | 密钥派生函数 | 日志输出函数 |
| 参数解析 | 需要手动过滤 label、读长度字节 | 直接拿到 label+secret+len，格式已就绪 |
| client_random 获取 | 需要结构体路径解引用 | 同样需要从 ssl_ptr 读取 |
| 对 SHA-384 的处理 | 需要动态读长度字节 | secret_len 直接作为参数传入 |
| 抗编译变体能力 | 强（依赖 RFC 标签） | 中（依赖函数本身未被内联/删除） |
| 发现函数的方法 | TLSKeyHunter 自动化 | 搜索 NSS 格式字符串的 XREF |

### 融合策略

两种方案互为补充，建议数据库同时记录两个 RVA：

```json
{
  "chrome": {
    "version": "130.x.xxxx.xx",
    "platform": "linux_x86_64",
    
    "hook_hkdf": {
      "rva": "0x048837E0",
      "fingerprint": "55 48 89 E5 41 57...",
      "note": "Derive-Secret，TLSKeyHunter方法，抗编译变体"
    },
    
    "hook_ssl_log_secret": {
      "rva": "0xXXXXXXXX",     // 待提取
      "fingerprint": "...",
      "note": "ssl_log_secret，BoringSecretHunter方法，实现最简单"
    },
    
    "hook_prf": {
      "rva": "0x0A22D4B0",
      "fingerprint": "55 48 89 E5 41 57..."
    }
  }
}
```

运行时策略：**优先使用 ssl_log_secret**（更简单可靠），如果找不到（极少数剥符号或特殊编译情况）则 fallback 到 HKDF hook。

### ssl_log_secret 的 Ghidra 定位方法

在 Ghidra 里搜索字符串 `CLIENT_HANDSHAKE_TRAFFIC_SECRET`（NSS 格式，不是 RFC 格式），其 XREF 指向的函数就是 `ssl_log_secret`。这个函数：
- 以 `CLIENT_` 或 `SERVER_` 开头的长字符串作为参数
- 会调用文件写入相关函数（如果 SSLKEYLOGFILE 已设置）
- 参数签名：`(SSL*, char*, uint8_t*, size_t)`，四个整数寄存器参数，极易识别

### 对 P8 数据库自动化阶段的影响

融合后数据库构建流水线变成：

```
输入: chrome二进制
    ↓
TLSKeyHunter (自动)
    → PRF RVA + fingerprint
    → HKDF RVA + fingerprint
    ↓
BoringSecretHunter方法 (自动，搜索NSS标签)
    → ssl_log_secret RVA + fingerprint
    ↓
输出: 三个hook点 + client_random路径
```

三个函数的定位都可以通过字符串 XREF 自动化，P8 阶段的 Ghidra headless 脚本只需增加一个搜索模块。

## 路径二：从 ssl_ptr 直接读 fd（不需要 hook SSL_set_fd）

BoringSSL 的 `ssl_st` 结构体内嵌了 BIO 指针，BIO 里有 fd。可以在 HKDF onEnter 时直接读取，完全跳过 SSL_set_fd：

```
ssl_ptr → rbio（偏移待确认）→ BIO.num = fd
```

需要在 Ghidra 里确认两个偏移量。在 `FUN_049837e0`（HKDF 函数）的反编译里搜索对 ssl_ptr 的解引用链，或者直接在 Frida 里用探针内存扫描的方式找：

```javascript
// 在 HKDF onEnter 里临时加这段，找 rbio 偏移
function probeFd(ssl_ptr) {
    // 扫描 ssl_ptr 开头 512 字节，找值在 3-1023 之间的整数（合理 fd 范围）
    for (let off = 0; off < 512; off += 8) {
        try {
            const ptr_val = ssl_ptr.add(off).readPointer();
            if (ptr_val.isNull()) continue;
            // 尝试读 BIO.num（通常在 BIO 对象偏移 +0x20 附近）
            const maybe_fd = ptr_val.add(0x20).readS32();
            if (maybe_fd >= 3 && maybe_fd < 1024) {
                send({t:'dbg', v:`offset ${off} → ptr → +0x20 = fd=${maybe_fd}`});
            }
        } catch(_) {}
    }
}
```
